# Phase 11C Streaming Stability Fix

## 背景

当前流式对话出现“什么都没说，但助手回复被插话侦测反复打断、UI 在聆听中 / 说话中横跳”的问题。

## 复盘结论

问题不只是阈值过敏，主要是 11C 把本地能量 VAD 同时用于两个目的：

1. 流式模式持续上行音频。
2. TTS 播放中本地侦测插话并立即 stopPlayback + sendAbort。

在手机外放场景，即使 Android AEC / NS 已开启，麦克风仍可能收到 TTS 残余回声。只要残余回声连续跨过本地 VAD 阈值，就会被当作用户插话。

此外，当前 UI 状态也由本地 VAD 回调直接切到 Listening，再由 TTS 音频帧切回 Speaking，因此会出现“聆听中 / 说话中”横跳。

## 修复策略

参考 py-xiaozhi 的音频管线思路：输入流和输出流保持独立并行，输入音频持续编码上行，输出音频持续播放；不要用本地播放残余能量强行驱动 UI 状态或本地 abort。

本次修复：

- 流式对话继续保持持续收音和音频上行。
- 流式对话不再启用本地能量 VAD 插话侦测。
- 流式对话不再由本地 VAD 自动 stopPlayback / sendAbort。
- 自然对话 VAD 保留，尾静音仍为 900ms。
- 明确打断仍可通过 UI “打断”按钮完成。

## 覆盖文件

```text
app/src/main/java/com/er1cmo/xiaozhiandroid/audio/AudioEngine.kt
```

## 验收重点

1. 流式对话下，助手第一轮回复不应再被自己的 TTS 残余回声反复打断。
2. UI 不应再因为本地 VAD 在“聆听中 / 说话中”之间横跳。
3. 麦克风仍持续上行，日志仍能看到 Opus 上行帧持续增长。
4. 自然对话保持约 900ms 尾静音后自动 stop。
5. 需要打断时，可以先用“打断”按钮。

## 后续方向

若后续要恢复真正语音插话，建议不要使用简单能量 VAD 直接触发本地 abort。更稳的路径是：

- 由服务端 realtime / VAD 结果确认插话；或
- 引入回声参考信号和更严格的双阈值 / 持续时长 / 频谱一致性判断；或
- 在 TTS 播放期间只做极强置信度本地打断。
